# Codex 작업 규칙

## 1. 문서 목적
이 문서는 VSCode Codex가 이 저장소에서 작업할 때 반드시 따라야 하는 공통 작업 규칙을 정의한다.

이 프로젝트는 웹 기반 셰이더 플레이그라운드 구현 프로젝트이며, 구현 과정에서 일관성, 문서 우선, 작은 작업 단위, 한글 문서화 원칙을 매우 중요하게 다룬다.

---

## 2. 최우선 원칙

### 2.1 문서 우선
Codex는 작업 전에 반드시 아래 문서를 읽고 기준으로 삼아야 한다.

- `docs/shader_playground_guide.md`
- `docs/shader_playground_development_plan.md`
- `docs/shader_playground_wbs.md`
- `docs/codex_working_rules.md`

문서와 현재 코드가 충돌하면, 충돌 원인을 파악하고 현재 작업 범위 안에서 가장 보수적으로 수정한다.

문서에 없는 대규모 기능을 임의로 추가하지 않는다.

### 2.2 한글 작성
아래 항목은 모두 한글로 작성한다.

- 문서
- README
- 코드 주석
- 작업 요약
- 커밋 메시지
- TODO 설명
- 변경 이유 설명

식별자 자체를 반드시 한글로 만들 필요는 없지만, 사용자에게 보이는 설명과 프로젝트 문서는 한글이어야 한다.

### 2.3 작은 작업 단위
한 번에 너무 큰 범위를 구현하지 않는다.

반드시 다음 원칙을 따른다.

- WBS 기준으로 작은 작업 단위로 나누어 진행
- 현재 세션 범위를 넘는 기능 확장 금지
- 구조를 크게 바꾸는 리팩터링은 명확한 이유가 있을 때만 수행
- 여러 기능을 동시에 밀어넣지 않기

### 2.4 변경 이유를 남길 것
코드를 수정했다면 아래를 항상 설명할 수 있어야 한다.

- 왜 이 파일을 수정했는가
- 왜 이 구조를 선택했는가
- 어떤 문서 항목을 반영했는가
- 어떤 후속 작업이 필요한가

## 3. 프로젝트 구현 원칙

## 3.1 렌더링 백엔드
현재 1차 구현 대상은 WebGL2이다.

- WebGL2를 우선 구현
- WebGPU는 즉시 구현 대상이 아님
- 다만 향후 확장이 가능하도록 구조 분리는 고려

즉, 지금 당장 WebGPU 기능을 억지로 넣지 말고, 렌더링 추상화 가능성만 열어두는 방향을 우선한다.

## 3.2 셰이더 입력 시스템
사용자가 작성한 셰이더 변수명을 가능한 그대로 노출한다.

- 예약 이름 강제 금지
- 타입 기반 UI 자동 생성 우선
- 필요 시 메타데이터 주석으로 UI 정보 보강
- 엔진 내장 입력만 예약 처리 가능

예:
- `uTime`
- `uResolution`
- `uMouse`
- `uModel`
- `uView`
- `uProj`

`NoiseTex`, `MaskTex` 같은 이름을 사용자에게 강제하지 않는다.

## 3.3 모델 업로드 정책
모델 업로드 지원은 아래 우선순위를 따른다.

1. `.fbx`
2. `.obj`는 후순위 또는 추후 지원
3. `.gltf`, `.glb`는 추후 검토 대상

처음부터 모든 모델 포맷을 받으려 하지 않는다.

## 3.4 플레이그라운드의 성격
이 프로젝트는 완전한 노드 그래프 에디터가 아니라, 우선 아래 목표를 가진다.

- 코드 기반 셰이더 편집기
- 실시간 프리뷰 뷰포트
- 모델/텍스처 업로드
- 자동 생성 머티리얼 인스펙터

즉, 1차 목표는 상용 엔진의 Shader Graph 전체가 아니라 Material Inspector에 가까운 경험이다.

## 4. 작업 시작 규칙

Codex는 구현을 시작하기 전에 반드시 아래를 먼저 정리해야 한다.

1. 이번 세션에서 수행할 작업 범위
2. 관련된 WBS 항목
3. 수정 예정 파일
4. 리스크 또는 주의점

작업 범위가 모호하면 가장 좁은 범위로 해석한다.

---

## 5. 구현 규칙

## 5.1 기존 코드 존중
이미 존재하는 구조가 있다면 먼저 그 구조를 이해하고, 불필요하게 전체를 갈아엎지 않는다.

다만 아래 경우는 예외다.

- 현재 구조가 명백히 문서와 충돌하는 경우
- 최소 변경으로는 해결이 불가능한 경우
- 유지보수성이 크게 떨어지는 경우

이 경우에도 변경 이유를 명확히 남긴다.

## 5.2 과도한 추상화 금지
초기 단계에서 필요 이상의 복잡한 추상화는 피한다.

예:
- 아직 백엔드가 하나뿐인데 지나치게 복잡한 다중 백엔드 시스템
- 아직 한 종류의 에셋만 쓰는데 범용 에셋 플랫폼 설계
- 아직 쓰이지 않을 대규모 플러그인 구조

지금 필요한 만큼만 구현한다.

## 5.3 에러 처리를 무시하지 말 것
셰이더 플레이그라운드는 실패 상황을 잘 보여주는 것이 중요하다.

따라서 아래는 반드시 고려한다.

- shader compile error
- program link error
- texture load failure
- model parse failure
- 필수 attribute 누락
- WebGL context 문제

에러가 날 수 있는 코드는 가능한 한 사용자가 이해할 수 있는 형태로 표면화한다.

## 5.4 임시 구현도 흔적을 남길 것
빠르게 진행하기 위해 임시 구현을 하더라도 다음을 남긴다.

- 주석
- TODO
- 후속 정리 필요 항목
- 문서 또는 작업 요약에 기록

---

## 6. 문서화 규칙

구현과 함께 최소한 아래는 갱신 대상인지 항상 검토한다.

- `README.md`
- 관련 `docs/*.md`
- 새로 만든 핵심 모듈의 사용 설명
- 작업 결과 요약

문서화가 필요한데 생략하지 않는다.

특히 다음 경우는 문서 반영을 검토해야 한다.

- 폴더 구조 변경
- 주요 인터페이스 추가
- 실행 방법 변경
- 환경 설정 변경
- 사용자 사용 흐름 변경

## 8. 커밋 규칙

커밋 메시지는 한글로 작성한다.

권장 형식:

- `초기 프로젝트 구조를 설정한다`
- `WebGL2 렌더러 초기화를 추가한다`
- `셰이더 컴파일 오류 패널을 구현한다`

커밋은 가능한 한 의미 있는 작업 단위로 나눈다.

한 커밋에 너무 많은 unrelated 변경을 섞지 않는다.

---

## 9. 권장 작업 순서

특별한 지시가 없다면 아래 순서를 우선한다.

1. 프로젝트 초기 세팅
2. WebGL2 렌더러 기초
3. fullscreen quad 렌더링
4. 셰이더 컴파일/링크 시스템
5. 코드 에디터 연동
6. uniform reflection 기반 인스펙터
7. 텍스처 업로드
8. 기본 도형/모델 프리뷰
9. FBX 업로드
10. 저장/불러오기
11. 메타데이터 기반 UI 개선
12. WebGPU 확장 대비 구조 정리

---

## 10. 금지 사항

다음은 명시적 지시 없이는 하지 않는다.

- 문서 범위를 벗어난 대규모 기능 추가
- 전체 구조를 무리하게 재작성
- 사용자가 요청하지 않은 프레임워크 대전환
- 너무 이른 최적화
- 문서화 없는 대규모 변경
- 영어 커밋 메시지나 영어 설명 남기기
- 여러 스프린트 분량을 한 번에 구현하기

---

## 11. 권장 응답 스타일

Codex는 작업 중간/완료 시 아래 스타일을 유지한다.

- 짧고 명확하게
- 무엇을 했는지 먼저 설명
- 수정 파일과 이유를 구체적으로 설명
- 다음 작업 제안을 작게 제시
- 과장 없이 현재 상태를 정확히 설명

---

## 12. 세션 시작용 체크리스트

작업 시작 전 반드시 확인한다.

- [ ] 관련 문서를 읽었는가
- [ ] 이번 세션 범위를 이해했는가
- [ ] 관련 WBS 항목을 확인했는가
- [ ] 수정할 파일 범위를 예상했는가
- [ ] 범위를 넘는 확장을 하지 않도록 계획했는가

## 13. 세션 종료용 체크리스트

작업 종료 전 반드시 확인한다.

- [ ] 구현 내용이 현재 세션 범위 안에 있는가
- [ ] 변경 이유를 설명할 수 있는가
- [ ] 문서 반영 필요 여부를 검토했는가
- [ ] 임시 구현/TODO를 남겼는가
- [ ] 다음 작업을 작은 단위로 제안했는가
